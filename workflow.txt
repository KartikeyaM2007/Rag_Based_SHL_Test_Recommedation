PHASE 1: OFFLINE DATA ACQUISITION & CLEANING (DATA INGESTION)
-----------------------------------------------------------------------------------

1. THE WEB SCRAPER (`scraper/scrape_shl.py`)
   The pipeline begins with raw data collection from the SHL product catalog.
   * Multi-Level Parsing:
     - Catalog Scraping: Iteratively polls the dynamic catalog table using 
       `requests` and `BeautifulSoup` to fetch assessment cards, identifying test 
       names, full links, and support indicators.
     - Detailed Crawling: For each card, the crawler drills deep into the individual 
       landing pages to extract comprehensive job levels, explicit supported 
       languages, and full test descriptions.
     - Support Parsing: Uses CSS attribute matching to detect remote and adaptive 
       support icons ("yes" or "no" circles).
     - Regex Duration Extractor: Uses highly robust multi-pattern Regex to parse 
       human-written durations (e.g., "30 minutes", "15-20 mins") into clean integer 
       `duration_min` and `duration_max` fields.
   * Resiliency Mechanisms:
     - Implements a built-in retry circuit (3 attempts with a 1.5-second backoff) 
       to handle transient 502/503 gateway errors during scraping.
     - Outputs raw nested JSON to `data/shl_catalog_raw.json`.

2. DATA NORMALIZATION & SANITIZATION (`scraper/clean_catalog.py`)
   Refines scraped JSON into strict normalized data models.
   * Text Normalization: Normalizes non-ASCII text via `unicodedata.normalize("NFKC")` 
     and strips double whitespaces to ensure consistent vector comparisons later.
   * Code Expansion: Converts technical test type codes into clear text:
     - 'K' -> 'Knowledge & Skills'
     - 'P' -> 'Personality & Behavior'
     - 'C' -> 'Competencies'
   * Unique Identity Generation: Assigns stable IDs by SHA1 hashing the canonical 
     url or concatenated title/description. Writes to `shl_catalog_clean.json`.

-----------------------------------------------------------------------------------
PHASE 2: OFFLINE LLM-ENHANCEMENT & EMBEDDINGS (THE INDEX)
-----------------------------------------------------------------------------------

3. SYNTHETIC DATA ENRICHMENT (`llm/enrich_catalog.py`)
   Before indexing, the catalog undergoes an enrichment cycle using Gemini 2.5 Flash 
   to exponentially boost retrieval accuracy (widening the semantic surface).
   * Structured Schema Engine:
     Uses strict JSON response schema typing to instruct Gemini to output exactly:
     - skills: Technical and soft skill tags extracted directly from context.
     - synonyms: Alternative terminologies and industry phrasing for the test.
     - summary: A hyper-condensed, keyword-rich search-optimized summary.
     - synthetic_queries: Generates 5-8 hypothetical natural-language queries a 
       hiring manager might search to discover this test.
   * State Retention: Employs persistent intermediate checkpoints to allow resuming 
     without data loss if rate limits are encountered. Writes `shl_catalog_enriched.json`.

4. VECTOR EMBEDDING GENERATION (`embeddings/generate_embeddings.py`)
   Converts textual information into dense geometric math space.
   * Document Composition: Combines all textual contexts (Name + Type + Summary + 
     Description + Synthetic Queries) to create a rich, multi-perspective document text.
   * Vector Indexing:
     - Calls Gemini's `models/gemini-embedding-001` to embed each document into a 
       high-dimensional mathematical representation.
     - Saves the precomputed vectors as a highly performant compressed Numpy array 
       `embeddings_gemini.npy` along with a `meta_gemini.json` lookup file.

-----------------------------------------------------------------------------------
PHASE 3: THE RUNTIME ARCHITECTURE (ONLINE BACKEND PIPELINE)
-----------------------------------------------------------------------------------

When a user hits the API, the system acts as a 3-Stage hybrid pipeline orchestrated 
by `app/retriever.py`.

5. THE KEY MANAGER LAYER (`app/key_manager.py`)
   To enable production-level traffic on standard Gemini free keys, the system operates 
   an automated Key Manager singleton.
   * Safe Rotation: Loads 3 API keys concurrently. After every API query, the active 
     client index increments (`(idx + 1) % 3`) and enforces a throttle sleep (1s) 
     to seamlessly and completely dodge 429 Rate Limits.

6. STAGE 1: QUERY REWRITING & REASONING (`llm/query_rewriter.py`)
   Converts user phrasing into technical specifications.
   * Uses Gemini 2.5 Flash to read the raw user search string.
   * Rewrites the string using HyDE (Hypothetical Document Embeddings) to generate a 
     statistically ideal document match text.
   * Extracts strict structured filters in JSON:
     - SKILL: e.g., "Java"
     - SOFT: e.g., "collaboration"
     - JOBLEVEL: "Management"
     - DURATION: 45

7. STAGE 2: HYBRID DENSE-LEXICAL SEARCH (`embeddings/hybrid_retriever.py`)
   Performs two distinct searches simultaneously and fuses the scores:
   * Sub-Search A (Vector / Semantic):
     Passes the rewritten query back to the embedding model. Uses NumPy to perform 
     a lightning-fast Cosine Similarity matrix multiplication across all vectors 
     in `embeddings_gemini.npy` (Weights: 75%).
   * Sub-Search B (TF-IDF / Lexical):
     Applies Sklearn's `TfidfVectorizer` to find exact keyword overlaps on technical 
     jargon (Weights: 25%).
   * Metadata Boosting (Fine-Tuning):
     Dynamically inspects search candidates. If the metadata contains matches to the 
     extracted duration constraint or job level, the algorithm injects a dynamic 
     scalar boost (+0.12) to the final score!
   * Output: Passes the highly curated **Top 20 Candidates** to Stage 3.

8. STAGE 3: LLM RE-RANKER & REASONING ENGINES (`llm/reranker.py`)
   Acts as the final recruiter's critical filter.
   * Receives the user's intent and the Top 20 candidates.
   * Gemini 2.5 Flash carefully reviews each candidate in sequence, pruning non-viable 
     logical matches.
   * The model returns the strictly finalized sorted list of Top 10 recommendations.
   * Crucially, Gemini generates a human-readable `relevance_reason` explaining 
     *why* that test fits the user's criteria.

-----------------------------------------------------------------------------------
PHASE 4: API EXPOSURE & BACKEND HOSTING (RENDER)
-----------------------------------------------------------------------------------

9. FASTAPI ORCHESTRATOR (`app/main.py`)
   * Exposes lightweight, async endpoints `/recommend` and `/health`.
   * Uses Pydantic schemas for robust input validation of queries.
   * Implements permissive wildcard CORS configurations (`allow_origins=["*"]`) to 
     enable seamless communication with decoupled Frontends.

10. DEPLOYMENT & INFRASTRUCTURE (`render.yaml` & `start.sh`)
    * Hosted on Render's Linux platform using a custom `render.yaml` Blueprint.
    * Bypasses Windows permission anomalies via explicit shell wrappers (`bash start.sh`).
    * Dynamically forces `export PYTHONUTF8=1` to ensure zero Unicode errors when 
      outputting Emoji logging metrics in standard CLI logs.

-----------------------------------------------------------------------------------
PHASE 5: FRONTEND VISUALIZATION (VERCEL)
-----------------------------------------------------------------------------------

11. SERVER-SIDE ROUTING & SECURE PROXYING (`next.config.ts`)
    To maintain clean code hygiene and security, Next.js intercepts all calls to 
    `/api/recommend`.
    * Leveraging Next.js edge rewrites, it acts as a secure network-level reverse proxy, 
      forwarding the browser traffic to the configured `$env:BACKEND_URL` (Render) 
      transparently, protecting the exact backend URL signature from standard UI builds.

12. REAL-TIME STATE PROVIDERS (`components/ServerStatusProvider.tsx`)
    * Because Render Free Instances spin down after 15 minutes, the frontend includes 
      a intelligent active poller.
    * It continually pings `/api/health` in 2s increments.
    * Provides smooth visual states to the user ("Booting up...", "Connecting...", 
      "Online") so the client is never left staring at a broken interface during a 
      cold start wake-up cycle.

13. REACT RENDERING SYSTEM (`app/page.tsx` & `components/Search.tsx`)
    * Features modern high-contrast design styling utilizing TailwindCSS utility classes.
    * Utilizes state-driven tracking for multi-tier interaction states (Idle, Loading, 
      Results, Empty, Error).
    * Presents results via fully responsive cards, rendering duration times, 
      dynamic tags (e.g. "Personality & Behavior"), rich summaries, and AI-generated 
      reasoning explanations.
